1. Architecture at a glance

Kilter is a single-process Python web application. No external message queue, no separate scheduler daemon, no cache tier. This is deliberate — fewer moving parts for IT to audit, fewer failure modes for ops.

LayerTechnologyPurpose
Web frameworkFastAPI (Python)HTTP API + HTML pages
Application serverUvicornASGI server hosting FastAPI
DatabaseSQLite 3 (single file)All state — sessions, assignments, open items, users, audit log
SchedulerStdlib threading (daemon thread)Runs scan / close / alerts / extract on schedule
AuthMicrosoft Authenticator (TOTP)MFA — no passwords
File I/Oopenpyxl, stdlib xml.etree.ElementTreexlsx generation; SWIFT MT940/950 parsing; ISO 20022 camt.053/camt.054 XML parsing
FCUBS ingest (optional)oracledb Python driverDirect pull from Oracle — only needed when using scheduled Flexcube extract
SMTP (optional)Stdlib smtplibEmail SLA alerts

Filesystem layout under the install directory:

d:\Kilter\
├── app.py                  # FastAPI application
├── scheduler.py            # In-process job runner
├── recon_engine.py         # Match proposer (1:1 + splits + FX)
├── open_items.py           # Rolling ledger + carry-forward
├── scanner.py              # Intake folder watcher
├── ingest.py               # SWIFT + Flex file parsing
├── swift_loader.py         # MT940/950 .out parser
├── iso20022_loader.py      # camt.053/054 .xml/.txt parser
├── certificates.py         # Month-end certificate generator
├── sla.py                  # Teams / email dispatchers
├── db.py                   # SQLite schema + migrations
├── auth.py                 # TOTP / session tokens
├── extract_flex_xlsx.py    # Oracle puller (optional)
├── templates/              # Jinja2 HTML templates
├── messages/
│   ├── swift/              # Scanner intake — SWIFT statements (.out MT, .xml/.txt camt)
│   ├── flexcube/           # Scanner intake — Flex .xlsx files
│   ├── processed/          # Archived after successful ingest
│   └── unloaded/           # Files that failed to ingest
├── exports/                # Generated xlsx reports
├── uploads/                # Manual-upload landing zone
└── kilter.db             # The whole database

2. Server specifications

Minimum (dev / pilot)

CPU2 vCPU, x86-64
RAM4 GB
Disk20 GB SSD
OSWindows Server 2019+, Windows 10/11, or Linux (Ubuntu 22.04 / RHEL 9)
Python3.12

Recommended (production, up to ~100 registered accounts, 50k sessions/year)

CPU4 vCPU, x86-64
RAM8 GB
Disk100 GB SSD (database + 2 years of exports + archived messages)
NetworkOutbound 443 to Teams webhook; SMTP to your corporate mail relay; Oracle 1521 to FCUBS scan listener (if Flexcube)
Uptime targetBusiness hours primary; 24×7 fine because scheduler is idle outside work window
Why so modest? SQLite happily handles hundreds of millions of rows on a single SSD. The bottleneck for this workload is not database throughput — it's the Oracle round-trip during the Flexcube pull. Right-size by network, not compute.

When to scale up

  • > 500 registered accounts — SQLite is still fine but consider migrating the audit log and job_runs tables to a separate archive DB to keep main queries fast.
  • Concurrent analysts > 20 — upgrade Uvicorn to --workers 2 behind a reverse proxy and put SQLite in WAL mode (already default in the schema).
  • Multi-region / HA — out of scope for this build. Migrate to PostgreSQL + dedicated scheduler process; the code is ready for either, but contact the team for the transition plan.

3. Prerequisites & network

  • Python 3.12 on the host. python --version should print Python 3.12.x.
  • pip — standard with Python.
  • Write access to the install directory.
  • TCP 8000 open on localhost (or wherever you bind).
  • For Flexcube pull: Oracle 1521 egress to the FCUBS SCAN listener; DB credentials with read access to acvw_all_ac_entries_new and sttm_dates.
  • For email alerts: SMTP 587 egress to your corporate mail relay with STARTTLS support.
  • For Teams alerts: HTTPS 443 egress to *.webhook.office.com.

4. Installation from scratch

Steps assume a Windows host; Linux steps are identical apart from path separators.

# 1. Pull the Kilter repository to the install root
mkdir d:\Kilter
# (copy the source tree, or clone from your git host)

# 2. Install Python dependencies
cd d:\Kilter
python -m pip install --upgrade pip
python -m pip install fastapi uvicorn jinja2 openpyxl pydantic pyotp qrcode[pil] python-multipart

# 3. Optional: Oracle driver for scheduled Flexcube pulls
python -m pip install oracledb

# 4. Optional: smoke-test the Python install
python -c "import app; print('ok, routes:', len(app.app.routes))"

# 5. Initialise the database + start the server (development)
python -m uvicorn app:app --host 127.0.0.1 --port 8000 --reload

# 6. First-run enrollment token is printed to the console. Example:
#   ==================================================================
#     FIRST-RUN ADMIN ENROLLMENT
#     Open: http://localhost:8000/enroll?user=admin&token=abc123...
#   ==================================================================

5. Environment variables

All configuration is optional — defaults ship in the code. Override via the process environment.

VariableDefaultPurpose
FCUBS_USER(empty)Oracle user for the Flexcube pull job. Without this, the scheduled flex_extract job returns a clear diagnostic and stays idle.
FCUBS_PASSWD(empty)Oracle password. Mount via a secret manager rather than plain env var for production.
FCUBS_WORKERS24Parallel workers when calling extract_flex_xlsx.py from the CLI. The in-app scheduler runs serially.
FCUBS_OUTPUT_DIRflex_statementsWhere the extractor writes xlsx. The scheduled job overrides this to messages/flexcube/.

Database path is hardcoded to kilter.db next to db.py. Move it by editing DB_PATH in db.py and symlinking if you need.

6. First-run admin enrollment

On first startup, Kilter creates a single admin user and prints a one-time enrollment URL to stdout. Open it in a browser:

  1. Scan the QR code with Microsoft Authenticator.
  2. Enter the first 6-digit code to confirm enrollment.
  3. The enrollment token is consumed — navigate to /login and sign in as admin.
If you miss the console output, you can re-issue a token from another admin account via Users → Reset enrollment. If no admin is available, stop the service, delete the totp_secret and totp_enrolled_at fields on the admin row via sqlite3 kilter.db, and restart — the app will reissue a token.

7. Running as a service

Windows — NSSM

Wrap uvicorn with the Non-Sucking Service Manager so it survives reboots:

nssm install Kilter "C:\Python312\python.exe" "-m uvicorn app:app --host 127.0.0.1 --port 8000"
nssm set Kilter AppDirectory "d:\Kilter"
nssm set Kilter AppStdout "d:\Kilter\logs\stdout.log"
nssm set Kilter AppStderr "d:\Kilter\logs\stderr.log"
nssm set Kilter Start SERVICE_AUTO_START
nssm start Kilter

Windows — Task Scheduler alternative

If NSSM is not approved, create a task that runs at startup, action = python -m uvicorn app:app --host 127.0.0.1 --port 8000, working directory = the install root. Set "If the task fails, restart every 5 minutes" on the Settings tab.

Linux — systemd

# /etc/systemd/system/kilter.service
[Unit]
Description=Kilter reconciliation service
After=network.target

[Service]
User=kilter
WorkingDirectory=/opt/kilter
Environment="FCUBS_USER=KILTER_REPORT_GENERATOR"
EnvironmentFile=-/etc/kilter/secrets.env   # keeps passwords out of unit files
ExecStart=/usr/bin/python3.12 -m uvicorn app:app --host 127.0.0.1 --port 8000
Restart=on-failure
RestartSec=10

[Install]
WantedBy=multi-user.target

8. Reverse proxy & TLS

Kilter binds to 127.0.0.1:8000 by default. Put it behind your organisation's TLS certificate via IIS, nginx, or Apache so analysts hit https://kilter.example.local (replace with your internal domain) instead.

nginx sample

server {
    listen 443 ssl http2;
    server_name kilter.example.local;
    ssl_certificate     /etc/ssl/certs/kilter.crt;
    ssl_certificate_key /etc/ssl/private/kilter.key;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 120s;   # xlsx exports can take ~30s on large sessions
    }
}

The TOTP login flow works unchanged; session tokens are issued over HTTPS and marked HttpOnly+Secure by the reverse proxy when you enable the corresponding options.

9. Backups & disaster recovery

What to back up

  1. kilter.db — the whole application state. Single file, back it up atomically.
  2. exports/ — generated certificates + daily breaks workbooks. Optional but cheap.
  3. messages/processed/ — historic SWIFT + Flex inputs. Size-sensitive; rotate to archive storage monthly.

Atomic SQLite backup

Never copy an open SQLite file directly — use the built-in online backup API:

# Windows batch / Linux shell
python -c "import sqlite3; s=sqlite3.connect('d:/Kilter/kilter.db'); d=sqlite3.connect('d:/backups/kilter-%DATE%.db'); s.backup(d); d.close(); s.close()"

Schedule nightly via Task Scheduler / cron. Retain 30 daily + 12 monthly + 7 yearly snapshots in tiered storage.

Recovery

  1. Stop the Kilter service.
  2. Move the corrupt kilter.db aside (don't delete — keep it for forensics).
  3. Copy the most recent backup into place as kilter.db.
  4. Start the service. The scheduler auto-resumes and will re-ingest any files still in messages/swift/ and messages/flexcube/.

10. Upgrades

Kilter's schema uses additive migrations — new columns and tables are added on startup; existing rows are preserved. The process:

  1. Back up kilter.db.
  2. Stop the service.
  3. Replace the source tree with the new version.
  4. Install any new Python dependencies: pip install -r requirements.txt if provided.
  5. Start the service. Watch the log for "ALTER TABLE" lines — that's the migration running.
  6. Verify the login page renders and one session opens cleanly.
Destructive migrations (column drops, rename of existing data) are rare but do happen. They're always accompanied by a release note saying "take a backup first". If a release note is silent on migrations, it's additive-only.

11. Monitoring & log collection

Application health

  • GET / on an authenticated session is the liveness check.
  • GET /scheduled-jobs (admin) returns scheduler_running: true/false — the single best indicator of "is the automation actually happening?"
  • The audit log accumulates scheduler_run entries for every fired job — a gap > 30 minutes during business hours indicates the daemon has died.

Forwarding logs

Uvicorn writes to stdout/stderr. Point NSSM / systemd / Task Scheduler at a log directory, then forward to your central log platform (Splunk, Graylog, Loki).

What to alert on

SignalMeaningSeverity
scheduler_tick_error in audit_logScheduler loop hit an uncaught errorWarn
No scheduler_run in last 30 min (08:00–18:00 UTC)Daemon likely deadCritical
Any job_runs with status=error for 3 consecutive runsUpstream system down or credentials expiredWarn
Disk > 85% on the install volumeMessage archive or exports eating spaceWarn

12. Flexcube Oracle connectivity

The direct-from-Oracle extract is optional but is the strongest pitch differentiator vs Corona ("ops never touches a file"). To enable:

  1. Install the driver: pip install oracledb.
  2. Provision a read-only Oracle user with grants on fcubsawa.acvw_all_ac_entries_new, fcubsawa.sttm_dates, and fcubsawa.sttm_branch.
  3. Set env vars on the Kilter service: FCUBS_USER, FCUBS_PASSWD.
  4. Enable the Flexcube pull (pre-scan) job at Scheduler. First run will probe the connection; on success, daily extracts land in messages/flexcube/ at 05:45 UTC.
Do not embed production Oracle credentials in systemd unit files or Windows scheduled task definitions. Use a secrets file with restricted ACL, or integrate with the bank's secret manager (CyberArk, HashiCorp Vault). The app reads from the process environment only.

13. Troubleshooting

SymptomLikely causeFix
Login screen loops back after entering codeClock drift on the serverSync NTP. TOTP tolerates ±30 s; more than that and codes never validate.
Scheduler shows red dot on admin pageDaemon thread died on startupCheck service log for the startup exception. Restart the service.
PermissionError: [Errno 13] on scanmessages/ directory ACLEnsure the service account has read+delete on messages/swift/ and messages/flexcube/.
SWIFT file shows up as "no_partner"No matching Flex file in the same scan windowExpected for shared-correspondent accounts; wait for the Flex file, or upload manually.
Ingest crashes with UNIQUE constraint failedDuplicate file re-dropped into messages/Files are deduplicated by SHA-256; look in messages/unloaded/duplicate/.
"Error binding parameter" in job outputMisconfigured custom job params_jsonValidate JSON in the job editor; empty is fine.
Reconcile engine runs slowVery large session (10k+ rows) exercising the 1:1 proposer's O(N×M) scanRaise min_ref_len tolerance to prune early; split statements upstream.

14. IT readiness checklist

  • ☐ Python 3.12 installed; python --version verified.
  • ☐ Install directory created with write access for service account.
  • ☐ Python dependencies installed.
  • ☐ Service registered (NSSM / systemd / Task Scheduler) and starts on boot.
  • ☐ Reverse proxy + TLS configured; HTTP → HTTPS redirect in place.
  • ☐ Backup job created and tested with a recovery drill.
  • ☐ Log forwarding active; scheduler-dead alert configured.
  • ☐ Oracle read-only user provisioned, tested; env vars set.
  • ☐ Teams webhook URL created + tested via SLA alerts → Dry-run.
  • ☐ SMTP relay access verified from the host.
  • ☐ Admin account enrolled; second admin enrolled as break-glass.
  • ☐ Ops & Internal Control users enrolled; MFA tested.
  • ☐ Runbook added to internal wiki linking to this manual.
