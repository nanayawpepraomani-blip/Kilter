1. What the Audit role sees

Audit is the most read-wide, write-narrow role in Kilter. Specifically:

  • Read every session, assignment, open item, certificate, user, role assignment, and audit log entry.
  • Read every config — cash accounts, tolerance rules, auto-categorisation rules, auto-grouping rules, scheduler jobs, SLA channels.
  • Write: nothing. There is no operation an audit user can perform that mutates state.

This intentional asymmetry makes it safe to hand an audit login to external reviewers for spot checks — they cannot accidentally disturb production state.

2. The activity log

Path: Admin → Activity log (also visible to Audit and Internal Control).

The activity log is an append-only record of every state-changing action in Kilter. Nothing in here is ever edited or deleted. If an error is recorded, the correction is a new entry — the original stays.

Entry shape

FieldMeaning
idAuto-increment primary key. Monotonic, so ordering by ID is chronological.
timestampISO 8601 UTC — 2026-04-23T21:50:11.714232.
actorUsername of the user (or system / system_scheduler).
actionMachine-readable action name — see action catalogue below.
session_idSession context if applicable; NULL for system-wide events.
detailsJSON blob — action-specific payload (amounts, IDs, reasons).

Action catalogue (abridged)

ActionWhen it fires
session_createdNew session ingested.
decision_confirm / decision_rejectAnalyst decided an engine-proposed match.
manual_match / manual_splitForce-match via the Register or Split view.
session_closedSession transitioned to closed (manual or auto).
open_item_auto_clearedCarry-forward engine matched a prior open item against a new session.
open_item_manual_clearedAnalyst hand-matched an open item.
open_item_written_offTerminal write-off. reason in details.
certificate_created / certificate_prepared / certificate_reviewed / certificate_signedCertificate lifecycle transitions.
user_created / user_updatedIdentity changes.
account_created / account_updatedRegistry changes.
tolerance_updatedPer-account tolerance changed.
scheduler_runScheduler executed a job. Details: job name, type, status, duration, output.
sla_alert_logSLA log-only channel fired.

Filtering

  • Actor — dropdown populated from distinct usernames in the log.
  • Action — dropdown populated with per-action counts so you see which actions dominate.
  • Session ID — narrow to one session's entire lifecycle.
  • From / to date — ISO date range (UTC).
  • Search details — free-text LIKE match against the JSON details + action.

The summary strip at the top gives a total-entries / distinct-actors / distinct-actions feel so you can calibrate sampling effort.

3. Drilling into a single break

Every open item has a History button that opens a timeline:

  1. Go to Open items.
  2. Filter to the account / status / age you're investigating.
  3. Click History on the row.

The drawer shows a chronological timeline:

  • Opened — when the item became an open item, from which source side, which session row.
  • Comment(s) — every analyst note posted on this item, with author + timestamp.
  • Audit events — every open_item_* action from the activity log that references this item.
  • Cleared / Written off — terminal event with actor + method (auto_carry / manual_match / write_off).

This timeline is the single-page story of a break. Give it to an auditor who's asking "what happened to this amount?" and they have their answer.

4. The certificate chain of evidence

A signed certificate is the head of a chain that leads back to every raw transaction. The chain is:

Certificate (signed, with snapshot_json)
    ↓
Session(s) for the account in the period
    ↓
Assignments (confirmed = matches in the figures; pending = ignored)
    ↓
SWIFT rows + Flex rows (per-transaction detail)
    ↓
Ingested files (SHA-256 hashes, original filenames, ingested_at, ingested_by)

To produce this for an auditor:

  1. Get the certificate ID from Certificates.
  2. Open Activity log, filter action = certificate_signed, find the matching entry. details.certificate_id links back.
  3. Download the certificate xlsx (which embeds the frozen snapshot) and the PDF (for the human-readable form).
  4. Filter action = session_created over the certificate's period and the specific account_id. Each entry has swift_sha256 and flex_sha256 — these are the provable inputs.
  5. Filter session_id = X for each session. You now have every confirmed/rejected decision with actor + timestamp.

Every step is query-able via the web UI and export-able via CSV (section 5).

5. CSV evidence export

The Download CSV button on Activity log exports the currently-filtered result set as CSV. Same filters as the page; no row limit.

Typical exports:

  • Full year's audit log (from_date = Jan 1, to_date = Dec 31, limit unbounded).
  • All certificate signings for the year (action = certificate_signed).
  • All write-offs with amount > threshold — filter on action open_item_written_off, then post-filter in Excel using the JSON details column.

Column order: id, timestamp, actor, action, session_id, details. The details column is raw JSON — use Excel's Data → Text to Columns or Power Query for structured analysis.

6. Raw SQLite access

For the ultimate "give me everything" case, the whole application state is in kilter.db — a standard SQLite 3 file. Read-only access is easy:

# Open read-only so nothing can be mutated
sqlite3 "file:d:/Kilter/kilter.db?mode=ro" "SELECT COUNT(*) FROM audit_log;"

All tables are documented in db.py's schema. Key ones:

  • audit_log — the activity log (append-only).
  • sessions, swift_txns, flex_txns — per-session data.
  • assignments — every match decision, with source (engine / manual / auto_carry / split / manual_split) and split_group_id where applicable.
  • open_items — the rolling ledger.
  • reconciliation_certificates — certificate records with snapshot_json for signed ones.
  • users, user_sessions — identities and login history.
Request read-only credentials from IT — don't run queries against the production file directly while the service is writing. Either point at a backup snapshot or open with mode=ro as shown above.

7. Common audit queries

"Who signed every certificate last year?"

SELECT rc.id, rc.period_start, rc.period_end, a.label AS account,
       rc.signed_by, rc.signed_at
FROM reconciliation_certificates rc
JOIN accounts a ON a.id = rc.account_id
WHERE rc.status = 'signed'
  AND rc.signed_at BETWEEN '2025-01-01' AND '2026-01-01'
ORDER BY rc.signed_at;

"Every write-off over $10,000 in the last quarter"

SELECT oi.id, a.label, oi.amount, oi.cleared_at, oi.cleared_by,
       oi.write_off_reason
FROM open_items oi
JOIN accounts a ON a.id = oi.account_id
WHERE oi.status = 'written_off'
  AND oi.cleared_at >= '2026-01-01'
  AND ABS(oi.amount) > 10000
ORDER BY ABS(oi.amount) DESC;

"Every force-match by a specific analyst"

SELECT al.timestamp, al.session_id, al.details
FROM audit_log al
WHERE al.action IN ('manual_match', 'manual_split')
  AND al.actor = 'jdoe'
ORDER BY al.id DESC;

"Which accounts had tolerance changes last year?"

SELECT timestamp, actor, details
FROM audit_log
WHERE action = 'tolerance_updated'
  AND timestamp >= '2025-01-01'
ORDER BY id;

"Scheduler job failures in the last 30 days"

SELECT jr.started_at, sj.name, jr.status, jr.duration_ms,
       substr(jr.output, 1, 200) AS output_head
FROM job_runs jr
JOIN scheduled_jobs sj ON sj.id = jr.job_id
WHERE jr.status = 'error'
  AND jr.started_at >= datetime('now', '-30 days')
ORDER BY jr.started_at DESC;

8. Regulatory & external audit scenarios

Scenario A: External / central-bank auditor arrives

You're asked to demonstrate that reconciliation is being performed for a specific nostro account.

  1. Create a temporary audit-role user for the auditor via Admin (expires in 30 days).
  2. Show them Cash accounts filtered to the account — evidence that it's registered.
  3. Show Sessions filtered to that account for the past year — evidence of daily reconciliation.
  4. Show Certificates for the account — evidence of monthly sign-off.
  5. Open one certificate, click PDF — signed artefact with maker / checker / approver stamps.
  6. Export the audit log CSV for the relevant period.

Scenario B: External audit confirmation of a single balance

Auditor picks one account + one date and asks for evidence of the reconciled balance.

  1. Find the session closest to the date at Sessions.
  2. Open the session review — the opening and closing balances are on the header panel.
  3. Open the Register tab — every SWIFT row and every Flex row with its final status.
  4. Export via the xlsx download: a per-session workbook with all the detail.

Scenario C: Internal audit of write-off appropriateness

  1. Activity log → action open_item_written_off, date range + actor.
  2. For each entry, click through to the open item's history (from Open items).
  3. Sample the reasons — does each write-off have a plausible rationale and prior engagement (comments)?
  4. Export findings.

Scenario D: Investigating a suspected control failure

  1. Identify the suspect period + user.
  2. Activity log filtered to that actor + period — scan for rapid sequences (reject, then manual-match).
  3. For every affected item, open the History drawer.
  4. If the pattern is consistent, export the audit log CSV and hand to internal audit for deeper review.

9. Retention policy

Kilter defaults preserve everything forever. Specifically:

  • audit_log — no retention. Every row from day one is kept.
  • job_runs — last 20 per job. Older runs are auto-pruned to keep the table fast; scheduler audit is also captured in audit_log, which is not pruned.
  • sessions, assignments, open_items — no retention.
  • user_sessions — only active sessions; expired ones are purged.
  • messages/processed/ — filesystem retention is whatever your backup policy enforces. Generally retained at least as long as the certificate that references them plus 1 year.

If a retention policy requires deletion of data after N years (GDPR right-to-be-forgotten applied to a user, for instance), this is a manual DBA exercise. It is not exposed in the UI because audit integrity is usually considered to override individual retention limits in a regulated banking context.
