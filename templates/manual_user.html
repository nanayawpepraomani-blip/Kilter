1. Sign-in & MFA

Open the Kilter URL. You land on the sign-in page.

  1. Enter your username.
  2. Open Microsoft Authenticator on your phone. Tap the Kilter entry.
  3. Type the 6-digit code.
  4. Press Sign in.

Your session lasts until the configured timeout. If you get bounced to the login page mid-work, your session has expired — sign in again.

New phone? Ask an admin to Reset MFA on your account. They'll share a fresh enrollment URL; scan the QR in Microsoft Authenticator and you're back in.

2. The dashboard

After sign-in you land on Dashboard. Four stat counters up top:

  • Pending decisions — assignments across all open sessions that need your call.
  • Confirmed matches — lifetime count.
  • Open sessions — sessions with at least one pending assignment.
  • Registered cash accounts — the pairings the scanner knows about.

Three charts below:

  • Match activity · last 14 days — stacked bar chart. Green = confirmed, red = rejected, amber = still pending. Hover a bar for the date + counts.
  • Ageing of open items — horizontal bars across 0-1d / 2-3d / 4-7d / 8-30d / 30d+ buckets.
  • By functional team — PSC TROPS / TREASURY / ... open-item counts.

If any open item is more than 30 days old, a red at-risk callout appears above the stats. Click its "Review now" button to jump straight to the aged-items filter.

Below the charts, the Recent sessions table shows the last 10 sessions with per-status counts. Click any session ID to jump to its Review page.

Presenter mode

For big-screen walkthroughs, open the command palette with Ctrl+K and pick Presenter mode (auto-refresh). The sidebar and topbar disappear, stats enlarge, and the page auto-refreshes every 60 seconds.

3. The intake flow — scan & manual upload

Files normally arrive automatically — your IT pipeline drops SWIFT statements into messages/swift/ (either .out MT940/950 or .xml/.txt ISO 20022 camt.053/054 — the scanner auto-detects) and Flexcube .xlsx files into messages/flexcube/. The scheduled Intake scan (every 15 min) pairs them and creates sessions. You don't need to do anything.

When you need to force a scan — files just dropped and you don't want to wait — click Scan now on the dashboard. Results appear as a table below the button:

StatusMeaning
ingestedFile parsed, paired, session created.
duplicateSHA-256 of the file already seen. Filed to unloaded/duplicate/.
unregisteredAccount identifier isn't in the cash accounts registry. Check Admin → Discovered accounts.
no_partnerSWIFT file arrived but no matching Flex file (or vice versa). Waits in intake for the partner.
mismatchFound a partner but currency or account key disagrees.

Manual upload

For one-off runs (historical replays, a file that the SFTP pipeline missed), use the Manual upload card: pick one SWIFT file and one Flexcube file, click Upload and reconcile. Deduplication + validation run identically to auto-scan.

4. Reviewing a session — Queue tab

Click any session from Recent sessions or from Sessions. You land on the Review page with three tabs: Review queue, Register, and Split view.

The header

  • Cash account, access area, currency.
  • Statement period — from the SWIFT statement's value dates.
  • Balances panel — opening, closing, difference, matched totals.

The queue card

Shows one pending assignment at a time. Two panels side by side:

  • Left — SWIFT — value date, amount, sign (C/D), our ref, their ref, booking texts.
  • Right — Flexcube — value date, amount, type (CR/DR), narration, trn_ref, external ref, module, user ID.

Above the panels, a coloured tier banner tells you how confident the match is:

TierMeaning
1 — greenReference found in narration + exact amount + correct sign. Confirm unless something looks off.
2 — amberReference matches but amount differs (typically FX rounding). Verify.
3 — orangeNo reference link; same value date; amount + sign match. Usually genuine — check the narration for confirmation.
4 — redAmount + sign match; dates differ by ≤1 day. Weakest tier; scrutinise.
5 — violet "split"1:N or N:1 aggregate match. Engine found multiple rows on one side summing to the single row on the other. Check the per-row numbers make sense.

Below the panels:

  • Amount difference banner — the Flex−SWIFT delta. Zero = great; non-zero = warn on FX or data error.
  • Competing candidates — other Flex rows that lost this assignment. If the primary looks wrong, check these.
  • Notes strip — prior notes on this match.

Acting on the queue

Three primary actions:

  • Confirm match — accept the engine's pairing. Writes decision_confirm to the audit log.
  • Reject — the pairing is wrong. Both rows go back into the unmatched pool. Audit: decision_reject.
  • Skip — come back later. No write.

Use C or Space to confirm, R or X to reject, S or J to skip forward, K to step back, N to add a note. Full shortcut list in section 13.

5. The Register tab

A flat table of every row in the session — SWIFT + Flex combined. Filters across the top:

  • Search — matches ref or description.
  • Status — all / confirmed / pending / rejected / unmatched.
  • Side — both / SWIFT / Flex.

Used for:

  • Global view — what's in the session, colour-coded by status.
  • Force-match — click checkboxes on one SWIFT row + one Flex row, click Force-match, optionally add a reason. See section 7.
  • Split-match — pick one row on one side and multiple on the other, the button label becomes Force split-match.
  • Split-group badges — violet "split" badges on rows that are part of an aggregate match.

6. The Split view tab

A two-pane side-by-side: SWIFT on the left, Flex on the right. Use this when the engine missed a match and you want to hunt for the counterpart visually.

Each pane has its own search + status filter and shows row counts + running sum of selected rows. Click a row to toggle selection. Multi-select on one side (while the other has exactly one selected) enables a split-match.

The amount-tie pill

Bottom-right shows a green Amounts tie ✓ pill when your selected-row sums match within tolerance, or a red Δ pill with the delta when they don't. When green, Match selected creates the assignment.

Keyboard flow

The split view is built for keyboard:

  • J / K move the focused row down/up in the current pane.
  • Tab switches between the SWIFT and Flex panes.
  • Space toggles selection on the focused row.
  • M matches the current selection.
  • Esc clears all selections.

7. Force-match & split-match

Used when the engine didn't propose a pairing but you know two rows belong together. Both sides must currently be unmatched — if a row is already on a pending or confirmed assignment, reject that one first.

1:1 force-match

  1. Open the Register tab (or the Split view).
  2. Tick the SWIFT row + the Flex row.
  3. Click Force-match. Review the preview banner ("Both rows are unmatched and amounts tie").
  4. Add an optional reason (audit trail).
  5. Click Match.

Split-match (1:N or N:1)

SWIFT posts one aggregate, Flex posts several (or vice versa), all with the same reference. Tick the single row on one side and all the partner rows on the other — the button relabels to Force split-match and shows 1:3 split ready (or similar) as a hint. Confirm.

The app creates N assignment rows sharing a split_group_id so the match can be undone or audited as a single unit.

8. Adding notes

Notes are free-text annotations on an assignment or an open item. Use them to leave context — "awaiting bank ref XYZ, expected T+2" — not final decisions.

  • On the Queue tab: press N or click the notes strip.
  • On Open items: click History on a row, then post a comment from the timeline view.

Notes are permanent — no edit, no delete. That's not a limitation, it's the audit trail.

9. Closing a session

When the queue is empty (or you want to stop for the day), click Close session on the review page. Kilter:

  1. Seeds every still-unmatched SWIFT and Flex row into the rolling open-items ledger.
  2. Auto-categorises each item via the active categorization rules.
  3. Auto-assigns each item to a functional team via the grouping rules.
  4. Stamps the session as closed, with your username + timestamp.

Closed sessions are immutable — the figures on the review page never change again. If more rows arrive for the same account later, they go into a new session; carry-forward matching may automatically clear today's open items.

You don't have to close manually. The scheduler's Auto-close stale sessions job runs daily at 23:00 UTC and closes everything older than 12 hours that you didn't get to. That's safe — unmatched rows go to the rolling ledger, not to the void.

10. Managing open items

Go to Open items. This is your rolling work queue — everything that's unmatched across every session you have scope for.

Filters

  • Account — all accounts or just one.
  • Status — open / cleared / written-off.
  • Category — filter by the auto-categorised break type.
  • Min age days — only show items older than N days.

Ageing badges

Each row has a coloured ageing badge: 0-7 (neutral) / 8-30 (info) / 31-90 (warn) / 90+ (danger).

Row actions

  • History — timeline drawer showing opened → comments → audit events → cleared/write-off. Great for auditors.
  • Clear — hand-match this item against a row in a current session. Pick the session ID + the row + which side.
  • Write off — terminal state for items ops has given up on. Requires a reason; goes into the audit log.

When to write off vs wait

  • < 30 days — usually wait. Most counterparts arrive within T+5.
  • 30-90 days — investigate. Contact the correspondent or the originator.
  • > 90 days — write off with a clear reason. Never just leave — aged items distort the ledger.

11. The daily breaks report

The multi-tab xlsx that replaces the manual daily breaks workbook. From the dashboard:

  1. Pick the As-of date (defaults to today).
  2. Click Download xlsx.

The file has tabs for:

  • Summary — overall counts and amounts.
  • BOG USD / EUR / GBP — central bank account breaks by currency.
  • TREASURY / TRADE SWITCHES / PSC TROPS / CSD FEES / GRA / CASH MGT / TPU — per-team tabs.

The scheduler's Daily breaks report job fires at 07:00 UTC every day and writes a marker into the audit log. The actual xlsx generates on-demand when you click Download — you always see fresh figures.

12. Preparing a certificate

Month-end sign-off artefact. You're the maker in the maker-checker-approver flow.

  1. Go to Certificates.
  2. Pick Account, Period start (first of month), Period end (last of month).
  3. Click Generate draft. A row appears in draft status.
  4. Review the figures — click PDF 🖨 to open the print-ready page. Sanity-check opening/closing balance, match counts, open-item totals.
  5. When ready, click Prepare. Optional note. Status becomes prepared.

Internal Control picks it up from there. You see the status update in real time when the page reloads. Once signed, the xlsx / PDF becomes the frozen month-end record.

14. Common situations & how to handle them

"The amounts tie but the dates are 5 days apart."

Likely a weekend or FX settlement lag. Confirm the match; leave a note: "T+5 settlement on public holiday". If it keeps happening, ask admin to bump this account's date_tol_days tolerance.

"SWIFT shows one debit for $10,000, Flex shows three credits for $3k/$3k/$4k."

Classic 1:N split. Use Split view: select the single SWIFT row, select the three Flex rows, M to match. If all three share a reference, the engine likely already proposed it as tier 5 — check the queue first.

"The engine matched row A to row B but row C is a better fit."

On the queue card, C should appear in the Competing candidates strip. Click Reject on the current proposal; the engine's next pass uses C.

"An open item has been sitting for 60 days."

Open it. Click History — have comments been posted? If not, post one now: what did you find out? If the counterpart genuinely isn't coming, write off with reason "counterparty unable to confirm".

"I accidentally rejected a good match."

It's not lost — the two rows go back to unmatched. Use force-match on the Register or Split view to re-pair them. The audit log keeps both actions (reject, then manual).

"My review queue has 200 pending items and I'm overwhelmed."

Use the tier filter: tier-1 matches are the easiest wins — high confidence, usually just confirm. Burn through those first with Space-J-Space-J. The harder tiers take attention but you'll see fewer.
