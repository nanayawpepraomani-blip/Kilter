1. What the Admin role can do

The Admin role has full write access to every configuration surface in Kilter. It is also the only role allowed to sign month-end certificates — the approver leg of the maker-checker-approver flow.

Admin is NOT the same as Ops. An admin who also runs the reconciliation queue should use a separate Ops-role account for the daily review work, keeping an audit-log clean separation between configuration changes and operational decisions.

Two admins, always. Provision at least two admin accounts so the "break glass" case (primary admin locks themselves out, leaves the team) has a path back. MFA reset requires an admin; only an admin can create an admin.

2. User management

Go to Admin → Users.

Creating a user

  1. Click + New user. Fill in username, display name, role.
  2. The app generates a one-time enrollment URL. Copy it to the clipboard and share it with the user over a secure channel — internal chat, signed email, or hand-delivered.
  3. The user opens the URL, scans the QR code with Microsoft Authenticator, enters the first 6-digit code. The token is consumed on first use.

Roles

RoleCan doTypical fit
adminEverything. Including sign certificates and manage users.Head of reconciliation, deputy head.
opsRun the reconciliation — scan, review, confirm/reject, clear open items, prepare certificates.Reconciliation analysts.
internal_controlRead all. Review (but not sign) certificates. Read the full activity log.Internal Control desk.
auditRead all. Read the full activity log. No write capability.Internal or external audit.

Deactivating a user

Users → row → Deactivate. Active sessions are revoked immediately; historic audit entries keep the username for the paper trail.

MFA reset

If a user loses their phone, go to Users → row → Reset MFA. A fresh enrollment token is issued. Share the new enrollment URL with them.

3. Access areas

Access areas group cash accounts by organisational unit — NOSTRO, AFFILIATES, regulator accounts, branch codes, etc. The taxonomy is configurable per deployment.

Every cash account belongs to zero or one access area. The topbar Active area picker lets each user scope all list views (sessions, open items, stats) to one or more areas. Users without a scope see everything.

An out-of-the-box list of common access areas is seeded on first run. Replace it with your own taxonomy at install time, or add new areas via the Cash accounts admin UI as they become relevant — don't bulk-delete existing ones, because historic sessions reference them.

4. Cash account registry

Cash accounts is the heart of the system: each row is a registered pairing of one SWIFT account (correspondent side) with one Flexcube GL (own side) in one currency.

Registering an account

  1. Click + New cash account.
  2. Pick the Access area. (Blank = ungrouped.)
  3. Enter a Shortname (e.g. "CITI NY VISA") and a Label (full descriptive name).
  4. Pick the Correspondent BIC from the Banks dropdown. If missing, click + Add bank inline.
  5. Pick the Currency from the managed list.
  6. Enter the SWIFT account (as it appears in the statement's :25: field) and the Flexcube GL.
  7. Save.
SWIFT account + Flex GL + currency is the unique key. The same SWIFT account can back multiple Flex GLs (shared-correspondent accounts) and will raise a duplicate error only if the full triple repeats.

Editing

You can update access area, shortname, label, BIC, notes, and the active flag. The identity fields (SWIFT account, Flex GL, currency) are read-only in the edit dialog — changing them would orphan historic sessions. If you genuinely need to change one, deactivate the old account and register a new one with the corrected identity.

Discovered accounts

When the scanner sees a SWIFT file whose :25: account isn't registered, it parks it in Admin → Discovered accounts with a count of how many times it's been seen. Review weekly:

  • Register if it's a legitimate account that slipped through the onboarding.
  • Ignore if it's noise (a test statement, a retired account). Ignored entries stop blocking scan.

5. Banks registry

Banks is the master list of counterparty BICs. Cash accounts reference this registry — strict enforcement means you cannot register an account with a BIC that isn't in the list.

Fields: BIC (8–11 alphanumeric), Name, Nickname, Origin (their = correspondent, our = your own bank's BIC), Type (bank / broker), Access area (optional), User code.

Seeded on first run with whatever BICs already appear in cash_accounts — you can safely rename the placeholders.

6. Currencies registry

Currencies manages the ISO 4217 code list that feeds the currency picker on cash-account forms. Seeded with the major and regional African currencies most pilots ask for (AED / AUD / CAD / CHF / CNY / DKK / EUR / GBP / GHS / JPY / KES / NGN / USD / XAF / XOF / ZAR / etc.).

Deactivating a code hides it from the picker but doesn't affect accounts that already reference it.

7. Exchange rates

Exchange rates is the FX registry that feeds the cross-currency tolerance branch of the match engine. Identity rows (GHS→GHS = 1.0, USD→USD = 1.0, etc.) are seeded automatically.

Adding a rate

  1. Click Add rate.
  2. Pick From / To currencies, enter the rate (amount in from × rate = amount in to).
  3. Set Valid from (default today) and Source (BOG, REUTERS, MANUAL).
  4. Save.

Adding a new rate for an existing pair automatically deactivates the previous active row — the engine always sees exactly one current rate per direction.

8. Tolerance rules

Per-account overrides for match engine thresholds. Admin → Tolerance & rules.

ParameterDefaultWhat it controls
amount_tol_abs0.01Absolute amount tolerance (any currency). 0.01 = match if amounts differ by a cent or less.
amount_tol_pct0.0Percentage tolerance. 0.5 = 0.5%. Activated in addition to abs.
date_tol_days1Date-mismatch tolerance for tier 4. 1 = same day or ±1 day.
min_ref_len6Minimum reference length for tier 1/2 to consider the ref. Below this the engine treats the ref as absent.
fx_tol_bps0FX spread tolerance in basis points. 25 = 0.25% after FX conversion at the current rate.

When to override

  • Accounts with lots of small fees where exact-amount match is too strict → bump amount_tol_abs to 0.05.
  • Cross-currency accounts where you want the engine to absorb FX rounding → set fx_tol_bps to 10–25.
  • Accounts with many weekend value-dated items → raise date_tol_days to 3.
Don't raise tolerances to hide problems. If an account starts producing many "just over tolerance" mismatches, investigate the upstream system — wider tolerance will hide a future real error.

9. Auto-categorization rules

Tag one-sided open items with a break category so ops know at a glance what's happening. First rule to match wins; order by priority ascending.

Break categories (fixed list): timing_diff, fx_mismatch, internal_journal, reversal_pending, bank_error, genuine_break, written_off, uncategorized.

Rule fields: side filter, narration-contains, type-equals, amount-min/max, category. Inspect the auto-applied category in the open-items grid; manual reclassification produces category_source='manual' and bumps the audit trail.

10. Auto-grouping rules (teams)

Assign each open item to the functional team that owns it — TREASURY, PSC TROPS, TRADE SWITCHES, CASH MGT, GRA, TPU, CSD FEES. Drives the per-team tabs in the daily breaks report.

Rules match on narration, ref prefix, Flex user ID, or type. Priority-ordered; first match wins. Default (lowest-priority) catch-all is PSC TROPS.

Seeded with common treasury / trade / cash-management user-ID and ref-prefix patterns on first run. Extend or replace with your bank's own patterns in Match patterns.

11. Scheduler

Scheduler shows every scheduled job, its next run time, last-run status, and last-run output.

Run now

Click Run now next to any job to execute it on-demand. Result is shown in an alert and added to the job's run history.

History

History opens a drawer with the last 20 runs — started-at, duration, status, output. Useful for post-mortems ("why did the 14:00 scan take 45 seconds?").

Editing schedules

Click Edit. Change cadence (every N minutes, or daily HH:MM UTC), toggle enabled, or provide job-specific params as JSON (e.g. {"min_age_hours": 6} to shorten the auto-close window).

12. SLA alert channels

SLA alerts manages the channels that the sla_check job fans out to when open items age past their threshold.

Channel kinds

  • Teams webhook — post a MessageCard to a Teams channel. Create the webhook via Teams → channel → Connectors → Incoming Webhook; paste the URL here.
  • Email (SMTP) — send an HTML summary via your corporate mail relay. Fields: host, port, user, password, from address, comma-separated to addresses, STARTTLS.
  • Log only — no external call; writes a summary to the audit log. Useful for staging and testing the flow without a live channel.

Per-channel settings

  • Threshold days — only items open longer than this trigger this channel. Default 30.
  • Access area filter (comma-separated list). Blank = all areas. Use this to send TREASURY breaks to one channel and PSC TROPS breaks to another.

Testing

Use Dry-run first — it runs the same SQL but doesn't actually post. The result alert shows the item count. Once happy, Send actually dispatches.

13. Match patterns & auto-rule promotion

Match patterns mines the audit log for narration tokens, ref prefixes, user IDs, and modules that correlate with confirmed matches.

Rows with confirm rate ≥ 80% and at least 3 samples show a Create auto-rule button. Click to open a prefilled modal:

  • Rule family — Auto-grouping (team assignment) or Auto-categorization (break type).
  • Match kind — narration-contains or ref-contains, pre-selected based on where the pattern was found.
  • Pattern value — pre-filled with the token; edit freely.

Review the generated rule name + priority, pick the target team or category, and save. The rule fires from the next scan forward.

Review the patterns quarterly. User turnover, new products, and changed correspondent formats all shift the pattern distribution. A rule that was 95% accurate in Q1 may be 60% by Q4.

14. Signing certificates

Admin is the only role authorised to sign a certificate. The full flow is:

  1. Ops generates a draft certificate at Certificates.
  2. Ops clicks Prepare → status becomes prepared. (The "maker".)
  3. Internal Control reviews and clicks Review → status becomes reviewed. (The "checker".)
  4. You click Sign → status becomes signed. The live figures are frozen into the certificate's snapshot JSON at this moment. Later ledger edits cannot rewrite history.

Any transition writes an audit_log entry with your username, the UTC timestamp, and an optional note. Reviewers can produce the full certificate chain for compliance on demand via the audit manual's evidence-export flow.
