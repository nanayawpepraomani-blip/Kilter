The solution

Kilter

What it is

A reconciliation service that runs itself.

Ingests your SWIFT and core-banking feeds automatically. Pairs transactions through five tiers of intelligent matching. Carries unresolved items forward as a rolling ledger until they clear or are written off. Produces the regulator-facing breaks workbook and month-end sign-off certificates on demand. Every touch is audit-logged.

Built in-house on Python and SQLite. Zero vendor lock-in. Auditable end to end.

Differentiator 1

The ledger outlives the session.

Open items carry forward until they clear or are written off. Breaks don't disappear at end-of-session — they stay on the ledger, visible, aged, and assigned to a team.

Legacy tools today

  • Break detected this session.
  • Session closes; break list is lost.
  • Tomorrow: new session, same break re-detected.
  • Analyst matches it again. And again. And again.

Kilter

  • Session closes; break moves to the rolling ledger.
  • Tomorrow: counterpart arrives; carry-forward auto-clears it.
  • Analyst never sees it. Audit log records the auto-match.

Differentiator 4

Intelligent matching beyond 1:1.

Five-tier engine

  • Tier 1 — ref match + exact amount
  • Tier 2 — ref match + FX-tolerated amount
  • Tier 3 — same day + amount + sign
  • Tier 4 — ±1 day + amount + sign
  • Tier 5 — 1:N / N:1 split aggregates

Smart features

  • Reference normalisation — ABC/12-3, ABC-12-3, 000ABC123 all collapse to one match.
  • FX tolerance — basis-point spread across a live FX-rate registry.
  • Split matching — 1 SWIFT row = 3 Flex rows summing. Ref-gated.
  • Learned patterns — audit-log-mined auto-rules with one-click promotion.

Expected uplift over strict 1:1 matching: 10–15% on historic misses, without relaxing any tolerance.

Future-proof

Already speaking ISO 20022.

SWIFT is retiring MT940/MT950 on the FIN network. Every correspondent will move to ISO 20022 XML messages. Kilter accepts both today — drop an .out MT file or an .xml camt file into the intake folder and the engine doesn't care which it gets.

camt.053

End-of-day statement — the successor to MT940/950. Carries OPBD / CLBD balances, entries, remittance info. Parsed natively.

camt.054

Intraday debit/credit notification — near-real-time entries as they hit the nostro. Lets us reconcile within minutes of a SWIFT arrival.

Namespace-tolerant

Supports camt.053.001.02, .04, .08, .09 — every version in the wild. New versions land as an additive mapping, not a rewrite.

Legacy MT-only tools

MT-only. Require a vendor upgrade (with vendor licence fee) before the first correspondent flips.

Kilter

Parses both formats side by side. No workflow change for ops. IBAN and proprietary account identifiers both supported. Zero cutover risk when a correspondent flips.

Platform, not point tool

Three reconciliation streams, one operator UI.

A team trained on nostro can immediately work mobile money and cards. Same review queue, same audit log, same role model, same deployment.

Nostro / GL

SWIFT MT940 / 950, ISO 20022 camt.053 / 054 against the core-banking GL extract. The original nostro/GL workstream — 4-tier matching, 75–85% tier-1 auto in typical pilot data.

Mobile money

Operator B2W & W2B feeds — M-Pesa Safaricom, MTN MoMo (agent + B2W + W2B), Airtel Money, Telcel Cash. Pre-seeded operator profiles; bind to a wallet account and ingest same day. xlsx native — no save-as-CSV step.

Card scheme

Issuer / acquirer / switch settlement files — Visa, Mastercard, Verve, Cardlink (GhIPSS). N-way matching engine joins on scheme reference. PCI-DSS-out-of-scope storage by design.

Commercial implication: mid-tier banks typically run two or three separate tools for these streams. Kilter consolidates them into one licence at one mid-5-figure annual cost — replacing a six-figure SmartStream / Trintech / vendor-cards bill.

Cards module

PCI-DSS-out-of-scope by design.

Reconciling cards usually drags more of the bank's stack into PCI-DSS scope. Kilter's architecture is the opposite — three structural commitments keep the reconciliation system out:

1 · No full PAN, ever

Loaders mask at the parser seam into first6 + last4 only. The schema has no column that could hold a full PAN — there's no path to insert one.

2 · Sensitive Authentication Data refused

CVV, track data, PIN blocks (DSS §3.2) get rejected at ingest. Settlement files don't include them; this is defence-in-depth for the day a bank ops team accidentally exports from the wrong system.

3 · Free-text PAN-redacted

Merchant names and notes get a Luhn-validated PAN sweep on ingest. Any embedded full PAN becomes first6***last4 before persistence.

Commercial bottom line: one less system in PCI scope means a smaller compliance bill at your next QSA review. The N-way match engine joins records on scheme reference (Visa TRR, Mastercard Banknet, switch RRN), classifies as matched / mismatched / unmatched, and surfaces mismatches first so an operator works the highest-leverage cases without scrolling.

Under the hood

Boring technology. On purpose.

🐍 Python 3.12 + FastAPI

Industry standard. Auditable source tree. No runtime magic.

💾 SQLite single-file DB

Zero DBA overhead. Back up by copying a file. Hundreds of millions of rows comfortably.

🧵 Stdlib-only scheduler

No APScheduler, no celery, no Redis. One daemon thread, auditable in 300 lines.

🔐 Microsoft Authenticator MFA

No passwords stored. TOTP via pyotp — the same stack Microsoft uses internally.

📄 openpyxl for xlsx

The same library ops already uses to open the files. Formatting-compatible.

Zero vendor lock-in

Self-hosted on your infrastructure. Source available under the licence agreed at sign-up. Walk away any time, port to anything.
