1. Engine architecture

The match engine lives in recon_engine.py. It's a pure-function proposer — given a list of SWIFT rows and a list of Flex rows, it emits ranked match candidates without touching the database. Persistence happens in ingest.py, which calls the engine and writes the results.

The engine is format-agnostic. Whether the SWIFT rows originated from an MT940/950 .out file (parsed by swift_loader.py) or from an ISO 20022 camt.053/camt.054 .xml/.txt file (parsed by iso20022_loader.py), they reach the engine as the same dict shape — same keys, same normalisation. A single registered account can receive MT one day and camt the next and the engine doesn't care. This matters for the SWIFT MT-to-ISO 20022 migration: no engine changes are required as correspondents flip.

The engine runs in four passes:

  1. 1:1 proposer (propose_candidates) — enumerates every plausible pair, scoring each at its strongest qualifying tier (1–4).
  2. Resolve (resolve) — greedy assignment: strongest tier first, tightest amount diff as tiebreak. Losers attach to winners as "competing candidates".
  3. Split proposer (propose_splits) — looks at rows the 1:1 pass couldn't pair, finds 1:N / N:1 aggregates with shared references.
  4. Carry-forward (open_items.carry_forward_match) — matches prior open items on this account against still-unpaired rows in the current session.

The order matters. 1:1 runs first so confident single matches don't get consumed by speculative splits. Carry-forward runs last because it's cross-session — it should only touch rows that the current session couldn't pair internally.

2. The five match tiers

Every engine-produced match carries a tier — 1 (strongest) to 5 (split). The tier shows on the review queue banner and is stored on assignments.tier for every match.

TierRuleWhen you see it
1Ref in narration + amount within tolerance + sign mirrorCorrespondent posted our reference and the amounts match. The confident band — confirm unless something looks odd.
2Ref in narration + sign mirror, amounts outside toleranceRef matches but amounts differ. Usually FX (swap to FX branch if set up) or a partial posting.
3No ref link; same value date; amount within tolerance; sign mirrorReference was stripped or missing but the date/amount/sign all line up. Usually genuine.
4No ref link; amount within tolerance; sign mirror; dates within N daysWeakest engine tier. Date mismatch ≤ date_tol_days. Scrutinise — use the narration as corroboration.
5Split: 1:N or N:1 aggregate with shared normalised referenceOne correspondent statement row matches the sum of N Flex rows (or vice versa). Ref-gated — no amount-only splits.

Tiers below 5 are 1:1. Tier 5 is always a split and always involves 2–4 rows on one side.

3. Reference normalisation

The same counterparty reference can travel through multiple systems and emerge formatted differently: ABC/12-3 vs ABC-12-3 vs 000ABC123 vs ABC 123. Kilter's normalize_ref() collapses all of these to a single canonical form.

The rules

Applied in order:

  1. Strip whitespace and these punctuation characters: / \ - _ . , ; : | ( ) [ ] { } " '
  2. Upper-case everything.
  3. Strip leading zeros, keeping at least one character so "000000" → "0" rather than empty.

Examples

InputNormalised
ABC/123ABC123
  abc-123  ABC123
000123123
REF 123/456REF123456
OUR.REF:ABC-123OURREFABC123

Where it's applied

  • _ref_in_narration() — tier 1/2 ref lookups. Both the needle (SWIFT :20: ref) and the hay (Flex narration + external_ref + trn_ref) get a dual-pass comparison: raw uppercase first, then normalised. A hit on either version matches.
  • carry_forward_match — same dual-pass logic for prior-open-item lookups.
  • propose_splits — uses normalised refs as the bucketing key when grouping rows for subset-sum matches.

Why the short-ref fallback

After normalisation, a very short string (< 3 chars) is treated as "don't use normalised" — otherwise common fragments like "US" or "GH" would collide unrelated transactions. The engine falls back to the raw uppercase match in that case. For the legacy-path ref threshold, the min_ref_len tolerance (default 6) applies to the raw ref.

4. Amount tolerance — absolute, percentage, FX

Tolerance.amount_ok(swift_amt, flex_amt, fx_rate) is the central function every tier consults. Three checks in order; any pass returns True:

Absolute — amount_tol_abs

Pass if |flex − swift| ≤ amount_tol_abs. Default 0.01 — matches exact-cent amounts plus rounding.

Percentage — amount_tol_pct

Pass if (|flex − swift| / max(|swift|, |flex|, 1.0)) × 100 ≤ amount_tol_pct. Default 0 (disabled). Use when a specific account has inherent percentage-based adjustments (e.g. commission take-outs).

FX — fx_tol_bps (requires fx_rate)

Pass if (|flex − swift × fx_rate| / max(|swift × fx_rate|, |flex|, 1.0)) × 10_000 ≤ fx_tol_bps. bps = basis points; 100 bps = 1%.

Only the 1:1 proposer and carry-forward can supply an FX rate today — split matching does not cross currency.

5. Sign mirroring rules

SWIFT uses C (credit) / D (debit); Flexcube uses CR / DR. The engine only considers a pairing when the signs mirror:

MIRROR_SIGN = {'C': 'DR', 'D': 'CR'}

Meaning: a SWIFT credit matches a Flex debit, and vice versa. Same-sign pairs never match — that would be a bookkeeping error.

For splits, every row in the "many" side must have the same sign — no netting across CR/DR.

6. Date tolerance

date_tol_days (default 1) controls tier 4. The rule uses _days_between(d1, d2) which computes |d1 − d2| in calendar days.

Typical settings:

  • 1 (default) — tolerate a weekend or single-day lag. Good for USD / EUR accounts settling through bridge times.
  • 3 — tolerate long public holidays. Use for EUR accounts during Christmas week.
  • 5+ — problematic. If you need this much slack, the upstream is sending data too sparsely; investigate upstream first.

7. Split matching (1:N / N:1)

propose_splits runs on the rows the 1:1 proposer left unmatched. Strict constraints to avoid false positives:

  1. Ref gate is mandatory. No amount-only subset-sum matches — too many common amounts would create spurious matches.
  2. Subset size is capped at 4 by default. Beyond 4 the combinatorial explosion makes the proposer slow and the false-positive rate rises.
  3. All subset rows must share a normalised ref (or a ref that's a substring / superset of the single-side ref).
  4. All subset rows must share the same sign — no mixed CR/DR aggregates.
  5. Both directions are searched: 1 SWIFT → N Flex, AND N SWIFT → 1 Flex. Duplicates (the same Flex row being claimed by both) are de-duped.

The subset-sum search prefers smaller groups — it tries pairs first, then triples, then quads. This means "1:2 beats 1:3" when both are valid, which reduces the chance of over-splitting.

Storage of a split match

A 1:3 split creates three rows in assignments, each pointing at the single SWIFT row and one of the three Flex rows, all sharing the same UUID split_group_id. Each row's amount_diff is the total delta divided by the row count (for accounting purposes). The UI groups them for display via the split_group_id.

8. The resolution step

resolve() consumes the flat candidate list and decides who gets what:

  1. Sort candidates by (tier, abs(amount_diff)) — strongest tier first, tightest amount as tiebreak.
  2. Walk the sorted list. For each candidate:
    • If either the SWIFT row or the Flex row is already claimed → this candidate becomes a competing candidate on the winner. Surfaced in the review queue.
    • Otherwise → claim both rows, this candidate becomes an assignment.

What you never see: rows claimed by multiple assignments. The greedy pass is deterministic and produces a clean set of 1:1 pairings.

Anything that didn't get claimed goes into unmatched_swift / unmatched_flex, which feeds the split proposer next.

9. Carry-forward (sliding lookback)

The rolling-ledger equivalent of re-matching. carry_forward_match runs once per session ingest, after the main proposer has finished but before seed_open_items_for_session.

What it does

For every open item on this session's account (status = open, any age), try to pair it against the current session's still-unclaimed rows. Matching uses the same Tolerance object as the main engine, so ref normalisation, amount tolerance, and date tolerance all carry over.

Why it's not time-limited

A 90-day-old open item finding its counterpart today is exactly the value of the rolling ledger. Arbitrary age cut-offs (e.g. "only look back 30 days") would re-introduce the session-bound-break bug that Corona suffers from.

What gets recorded

A carry-forward clear creates an assignment with source='auto_carry' and open_item_id set, plus updates the open_item to status='cleared', cleared_via='auto_carry'. The audit log records open_item_auto_cleared.

10. Auto-categorisation & auto-grouping

Both are rule-lists evaluated in priority order; first match wins. They run at the moment an open item is seeded (end of session close) — not at match time, because matched rows never become open items.

Auto-categorisation

Tags the break type: timing_diff, fx_mismatch, internal_journal, reversal_pending, bank_error, genuine_break, written_off, uncategorized. Rule fields: side, narration-contains, type-equals, amount-min, amount-max, category.

Auto-grouping

Tags the owning team: TREASURY, TRADE SWITCHES, PSC TROPS, CSD FEES, GRA, CASH MGT, TPU. Rule fields: side, narration-contains, ref-contains, type-equals, amount-min, amount-max, functional_group.

Default priorities:

  • 10–19: individual Flex user IDs (most reliable).
  • 20–29: ref prefixes (deterministic for specific codes).
  • 30–39: narration keyword fallback (catches generic users doing team-specific work).
  • 99: catch-all default (PSC TROPS).

11. Tuning guide by symptom

SymptomLikely causeTuning
Many tier-1 matches get rejected The engine is wrong or the analyst is. Check 3 rejected cases. If the engine is wrong: your upstream ref is being reused (non-unique). Bump min_ref_len on that account to force higher-tier matches. If the analyst is wrong: coach.
High tier-4 rate with lots of skips Engine is grasping — dates misaligned. Reduce date_tol_days to 0 if same-day feeds are reliable — you'll fall to more unmatched but cleaner output.
Amount-off-by-a-few-cents misses Rounding in the upstream. Raise amount_tol_abs to 0.05 or 0.10 depending on currency and volume.
Cross-currency near-misses FX rounding over conversion. Populate fx_rates for the pair, set fx_tol_bps to 25.
Lots of splits where there shouldn't be Shared reference across unrelated rows. Bump min_ref_len. If that doesn't help, disable split matching for this account by setting a very high max_group_size=1 override (requires code change).
Split matches confirm then get unwound The engine found a false aggregate. Tighten normalisation or raise min_ref_len. Post-mortem on the false positive to understand the pattern.
Nothing matches after a format change upstream Reference format changed; normalisation no longer sees the common fragment. Check what ref the SWIFT is sending; if it dropped the prefix, update the ref_contains rules in auto-grouping to match the new format. Consider extending normalize_ref.

12. Match-pattern analytics

Match patterns is the engine's self-reflection. It mines decided assignments (confirmed + rejected) and computes:

  • Narration tokens — any ≥4-char uppercase word, excluding stopwords, seen in ≥ N assignments.
  • Ref prefixes — the first 4 characters of trn_ref.
  • Flex user IDs — the posting user from the Flex side.
  • Flex modules — the module that generated the entry (AR, FX, etc.).

For each key, you see the confirm rate — what fraction of decided assignments containing this key were confirmed vs rejected.

Promotion threshold

Tokens with confirm rate ≥ 80% and ≥ 3 samples show a Create auto-rule button. The promotion doesn't change engine behaviour — it creates an auto-grouping or auto-categorisation rule that fires on the next open-item seed.

What to read from the data

  • High confirm, high count — rock-solid pattern. Promote without hesitation.
  • Low confirm, high count — dangerous. A frequent pattern that often produces wrong matches. Consider bumping min_ref_len so the engine ignores this key.
  • Low count — insufficient data. Wait for more decisions before acting.

13. Proposer-to-ledger data flow

For debugging: what happens from file arrival to an open item?

1. scanner.scan() picks up a SWIFT file (.out MT940/950, or .xml/.txt camt.053/054)
   + a Flexcube .xlsx that share an account key.

2. ingest.ingest_pair():
   a. loaders parse the files into dicts (swift_loader for MT, iso20022_loader
      for camt — both emit the same txn + meta dict shape)
   b. account_meta.lookup_account_match() finds the cash_accounts row
   c. load_tolerance(account_id) builds the per-account Tolerance
   d. propose_candidates(swift_txns, flex_txns, tol=tol) → list[Candidate]
   e. resolve(candidates, ...) → Resolution (assignments + unmatched)
   f. INSERT INTO sessions, swift_txns, flex_txns, candidates, assignments
   g. propose_splits(..., unmatched_swift, unmatched_flex, tol=tol)
   h. INSERT INTO assignments for each split row, sharing a split_group_id
   i. carry_forward_match(session_id, tol) → re-try prior open_items against
      the session's still-unassigned rows. New confirmed assignments + the
      open_items get flagged cleared.
   j. seed_open_items_for_session(session_id): whatever's still unmatched
      becomes an open_item, auto-categorised + auto-grouped.

3. audit_log rows are written at every step: session_created, open_items_seeded,
   open_item_auto_cleared.

4. Ops walks in, reviews the pending assignments, confirms/rejects. Each
   decision writes an audit_log row and updates assignments.status.

5. At day-end the scheduler runs daily_close, which closes sessions with
   pending > 12h old. Final open_items become the rolling ledger.

Every step above is observable in the audit log and in the database. If a break has a story — "opened, commented twice, carried forward three times, cleared on day 47" — you can reconstruct every step from the data.
